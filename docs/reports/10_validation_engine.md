# Report 10 — Validation Engine

**Project:** Harmony Clinical Structuring (Project 3)
**Date:** 2026-05-30
**Branch:** feature/p3-schema

---

## 1. Overview

The validation engine (`app/ingestion/validator.py`) is a four-layer pipeline that
converts the raw string output of the fine-tuned model into a validated, structured
`ExtractionResult` Pydantic object. It is called by `ClinicalExtractor.extract()`
immediately after model generation and also used directly by the evaluation harness.

---

## 2. Design Principles

**Never raise.** The engine always returns an `ExtractionResult`. On any failure it
returns a valid but empty result with `validation` flags all set to False and an
`error_reason` string explaining the failure.

**Layered defense.** Four independent layers handle progressively stronger guarantees:
JSON syntax → JSON repair → schema validation → semantic evidence checking.

**System-injected fields.** `record_id`, `validation`, and `error_reason` are never
generated by the model — they are injected by the validator before Pydantic validates.
This keeps the model output surface minimal and separates system metadata from model output.

---

## 3. Validation Pipeline

### Layer 1 — Model Output (upstream)

The model generates a raw string targeting this JSON structure:

```json
{
  "schema_version": "v1",
  "entities": [
    {
      "entity_type": "medication" | "adverse_event",
      "mention": "<string>",
      "dosage": "<string>" | null,
      "linked_medication": "<string>" | null,
      "evidence": "<string>",
      "source_span": {"start_char": <int>, "end_char": <int>}
    }
  ],
  "relation_status": "related" | "not_related" | "none"
}
```

The model is explicitly instructed not to output `record_id` or `validation` blocks.

### Layer 2 — JSON Parse + Repair

```python
try:
    parsed = json.loads(raw_text)
    json_valid = True
except json.JSONDecodeError:
    json_valid = False
    parsed = json_repair.loads(raw_text)  # fallback
```

- `json.loads()` is attempted first. On success, `json_valid = True`.
- On failure, `json_repair.loads()` is attempted (handles truncated JSON,
  trailing commas, missing closing brackets, etc).
- If `json_repair` also fails or returns a non-dict, `build_empty_result(reason="json_parse_failed")` is returned.

**Observed rates (lora_v1 on 2,376 test examples):**
- Pre-repair validity: 100% (model never produced malformed JSON after fine-tuning)
- Post-repair validity: 100%

### Layer 3 — System Field Injection + Pydantic Validation

```python
parsed["record_id"] = record_id
parsed["validation"] = {
    "json_valid": json_valid,
    "schema_valid": True,
    "enum_valid": True,
    "evidence_present": True,
}
result = ExtractionResult.model_validate(parsed)
```

Both `record_id` AND `validation` are injected before `model_validate()` is called.
`ExtractionResult` requires both fields — passing the dict without them would raise
`ValidationError`.

On `ValidationError`: `build_empty_result(reason="schema_invalid")` is returned.
Enum enforcement is handled by Pydantic `Literal` types:
- `entity_type`: `Literal["medication", "adverse_event"]`
- `relation_status`: `Literal["related", "not_related", "none"]`

### Layer 4 — Evidence and Hallucination Checks

```python
for entity in result.entities:
    if entity.evidence and entity.evidence not in input_text:
        result.validation.evidence_present = False

    if entity.mention and entity.mention not in input_text:
        logger.warning("HALLUCINATION: mention '%s' not in input", entity.mention)
```

**Evidence check:** If any entity's `evidence` field is not a substring of `input_text`,
`validation.evidence_present` is set to False. The result is NOT rejected — it is
returned with this flag for downstream analysis.

**Hallucination check:** If any entity's `mention` is not a substring of `input_text`,
a warning is logged. The result is returned (not rejected) since the pipeline must
not crash on individual extraction failures.

## 4. build_empty_result

```python
def build_empty_result(record_id: str, reason: str) -> ExtractionResult:
    return ExtractionResult(
        record_id=record_id,
        schema_version="v1",
        entities=[],
        relation_status="none",
        validation=ValidationFlags(
            json_valid=False, schema_valid=False,
            enum_valid=False, evidence_present=False,
        ),
        error_reason=reason,
    )
```

| `reason` value | Trigger |
|---|---|
| `extraction_disabled` | `EXTRACTION_ENABLED=false` env var |
| `empty_input` | `text` is empty or whitespace-only |
| `model_load_failed` | `_ensure_loaded()` raised an exception |
| `extraction_error` | Inference call raised OOM/CUDA/network error |
| `json_parse_failed` | Both `json.loads()` and `json_repair` failed |
| `schema_invalid` | `model_validate()` raised `ValidationError` |

## 5. ExtractionResult Schema

```python
class ValidationFlags(BaseModel):
    json_valid: bool
    schema_valid: bool
    enum_valid: bool
    evidence_present: bool

class SourceSpan(BaseModel):
    start_char: int
    end_char: int

class Entity(BaseModel):
    entity_type: Literal["medication", "adverse_event"]
    mention: str
    dosage: Optional[str] = None
    linked_medication: Optional[str] = None
    evidence: str
    source_span: SourceSpan

class ExtractionResult(BaseModel):
    record_id: str
    schema_version: Literal["v1"]
    entities: list[Entity]
    relation_status: Literal["related", "not_related", "none"]
    validation: ValidationFlags
    error_reason: Optional[str] = None
```

**Key constraint (D-35):** `validate_extraction()` receives `input_text` as the
original (unmasked) clinical text. All `source_span` offsets are valid against this
string. PHI masking happens downstream in the ingestion pipeline and must never
precede extraction.

---

## 6. Integration Point

`validate_extraction()` is called in `ClinicalExtractor.extract()`:

```python
raw = self._generate(text, record_id=record_id)
return validate_extraction(raw_text=raw, record_id=record_id, input_text=text)
```

The `ClinicalExtractor` wraps the entire call in a `try/except` so even if
`validate_extraction` itself had a bug, the pipeline degrades gracefully.
In practice `validate_extraction` never raises.

---

## 7. Relations Construction

`relations[]` in the OpenSearch document (built in `app/api/documents.py`) is
derived from `linked_medication` on each `adverse_event` entity:

```python
# From adverse_event entities with a non-null linked_medication
relations = [
    {"adverse_event": ae.mention, "medication": ae.linked_medication}
    for ae in extraction.entities
    if ae.entity_type == "adverse_event" and ae.linked_medication
]
```

Relations are NOT constructed as a cartesian product of medications × ADEs —
they are only created when the model explicitly sets `linked_medication`.

## 9. Conclusions

The validation engine provides strong correctness guarantees for the extraction
output while maintaining a "never crash" contract with the ingestion pipeline.
After LoRA fine-tuning, the engine operates entirely in the happy path:
JSON validity is 100% pre-repair, schema validity is 100%, and hallucination
rate is 0.04% — meaning the repair and fallback layers are effectively idle
in production. They remain important for robustness against model regressions
or unusual input texts.
